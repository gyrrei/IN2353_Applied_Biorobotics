Assignment IX - Particle Swarm Optimization (PSO) Individual

Posted by Gyri Reiersen on 20 June 2018, 9:22 PM

Introduction

The Particle Swarm Optimization (PSO) was developed as by Kennedy and Eberhart [1] in 1995 as an optimization technique that mimics swarm behaviour in nature, such as birds or fishes. Compared to classical optimization methods, the PSO does not use a gradient and is more similar to random search, but with a memory of where other particles have found optimums.

The problem statement is as follows; “Implement the PSO algorithm (all topology) for the given model. Optimize up to 3 parameters. Design the algorithm in a way that allows for distributing model evaluations among team members”.

In this assignment, we will use the PSO to propose a method of finding the optimal parameters, \(PEA, AEA, G_{MH}\) for a stable gait of the JenaFox simulation model. We will be implementing an algorithm described by Simon [2] in order to get the highest velocity possible, maintaining stability. Later, we will in teams use the best performing algorithm of the team members and adapt it to run the optimization as parallel computing to reduce the computation time. Four student teams are to compete against each other to see who can get the best result, with the shortest computation time.

 

Approach

The paper [1] proposes a general algorithm for PSO, with some alternatives for the programmer to choose. Initially, there are three general dimensions of the algorithm, \(N\) the number of particles in the swarm population, \(G\) number of generations or iterations, and lastly \(D\) the number of parameters to be optimized. The first two are hyperparameters that improve the performance when set to a high value, but then also increase the computation time. The search space to be covered is of \(D\) dimensions and will therefore exponentially increase the complexity of the problem and the computation required when increased.

As we are implementing an all topology PSO, which means that the neighbourhood is essentially the entire population, we needed to define the best in the neighbourhood as the best in the population. Moreover, it is needed to initialize the particles with a uniform distributed along the dimensions of the search space. The update of the particles happens in two steps

The values for the hyperparameters are as follows:

Parameter space min-max

\(x_min = [0 -15 1]  x_max = [15 0  2]\)

Maximum learning rate

\(\phi_{max} = 2.05\)

As suggested in the paper, the learning rates \(\phi_1\) and \(\phi_2\) were chosen at random for each particle.

Initially, the parameter limits for \(PEA\) and \(AEA\) was set to +-90 degrees, but for the test populations with only 30 particles, this parameter space turned out to be too large, and no stable gait was found. By decreasing the parameters to +- 15, stable solutions were found more quickly and with higher performance.

The maximum velocity was initially set to 10% of the search space range but was later changed to 20% to increase the level of exploration in a combination of implementing a decreasing inertia weight to increase the level of exploitation throughout the generations.

Figure 1: The initialized particle swarm with the performance (velocity) illustrated by the colour scale

 

Figure 2: The particle swarm after 5 generations. You can see how the swarm is starting to converge towards an optimum

When the model was walking stably with 5 particles and 5 generations, it was time to improve the performance. We increased the number of generations to 20 and particles to 30 so that the search space was better covered. However, in order to improve the algorithm, it was necessary to run through the optimization a couple of times, each time taking between 4-7 hours.

The best result \(v_{max} = 0,897\) was obtained after ~4 hours of computing.

References

[1] Kennedy, Eberhart; Particle Swarm Optimization, IEEE, 1995

[2] Simon; Evolutionary Optimization Algorithms, Wiley, 2001

Assignment VIII - Neural Network Based Controller

Posted by Gyri Reiersen on 24 May 2018, 10:42 AM

Introduction

In order to control and achieve human-like locomotion, researchers have been inspired by the human control organ; the brain and its network of neurons. Although the human brain is complex, a neural network is able to simulate transmission of information in the electrical impulses of the neurons and learn patterns to make decisions based on training data. In this document, we will be presenting a neural network based controller that generates a stable gait for a simulation of a robotic bipedal walker. 

Problem Specification

The problem is defined as "You will program and explore the neural network based controller as described in Geng 2006. You will be provided with a sample dataset for testing, which will contain a time series of controller input data. Once the controller is generating useful outputs, you will try them on the multi-body simulation. Document your approach and code in Mahara. Comment on the challenges you discovered during implementation. Identify parameters that influence the robots average forward velocity. Comment on the challenges you discovered during implementation. Identify parameters that influence the robots average forward velocity.".  This means that we will implement a neural network based control based on the controller model in the paper [1] and the corresponding JenaFox robot model in Simulink. The paper [1] shows that a controller based on a neural network can in some cases be a better control solution to bipedal walking that other approaches, such as zero moment point or inverted pendulum because it can walk with a stable high speed. 

The Robot Model

For simulating the neural controller, we have been provided with a multi-body robot model in Simulink. This is a bipedal robot, with two actuated joints in each leg, a hip and a knee. In each joint, there are motors that adjust the angle and these angles are the sensory input for the controller. There is also a ground contact sensor in each foot. The movement of the robot is constrained to the two-dimensional x-y space. For testing the controller in the simulation, we have been provided with a sample dataset with time series of input data. Initially, a stable controller based on an equilibrium point was provided. The task was to change the existing controller with a neural network based one. The controller is implemented in MatLab as a function block and will be called during the simulation in Simulink. 

Figure 1: A schematic of the robot model and neuronal controller in Simulink 

Development of the Neuronal Controller

The sensor-driven neuronal controller will be implemented based on the description in Geng's paper [1] with some modifications. In figure 2, we can see the control parameters for the robot model taken from [1]. However, in our model, we will correct the convention of flexion and extension in the hip joint and the axis of reference as seen in figure 3. 

There are indices s (sensor) and m (motor), h (hip) and k (knee), l (left) and r (right), as well as e (extensor) and f (flexor) used in this paper to specify the different parameters.

Figure 2 : Control parameters for the joint angles, taken from [1]

Figure 3 : Definitions and coordinate system for our model 

Here is an overview of all the controll parameters of the model:

Parameters for Sensory Neurons  Values Unit
\(\alpha_{ES(FS)}, \alpha_{AL(AR)}\) 2  
\(\theta_{ES,h}\) 5 deg
\(\theta_{FS,h}\) -10 deg
\(\theta_{ES,k}\) -3 deg
\(\theta_{FS,k}\) -80 deg
Parameters for Motor Neurons Values Unit
\(\tau\) 0.01 s
\(M_{AMP}\) 3   
\(G_{Mh}\) 1.5   
\(G_{Mk}\) 0.9* \(G_{Mh}\)   
Table 1 : Controll parameters of the model

The Modelling of the Neural Network

In this model, we will be using two types of neurons; sensory neurons and motor neurons, as well as stretch receptors and ground contact sensors. The top-level structure of the network is shown in figure 4. The network is adapted to our model by exchanging the motor neurons and the respective sensor neurons in the hips and with changes to the stretch receptors.

Figure 4: The neuron model of the controller. The small numbers give the values of the connection weights.
(Adapted to our model from [1] )

Stretch Receptors

To model sensory neurons, we often use the sigmoid function as the activation function because it is limited between zero and one. In our model, the stretch receptors are signal the attainment of the AEP (Anterior Extreme Position), the threshold for the flexion of the hip. If the hip angle exceeds the threshold of \(\theta_{AL}\) or \(\theta_{AR}\) the respective receptor is activated and the output of the sigmoid is changed rapidly from 0 to 1. As we exchanged the extension and the flexion convention of the hips, the threshold is defined as the AEP for flexion in out model. 

\(\theta_{AL} = \theta_{AR} = \theta_{HF} \)

\(\rho_{AL} = (1 + \exp^{\alpha_{AL}(\theta_{AL}-\phi)})^{-1} \)

\(\rho_{AR} = (1 + \exp^{\alpha_{AR}(\theta_{AR}-\phi)})^{-1} \)

Ground Contact Sensors

The ground contact sensors are activated when the foot is in contact with the ground and outputs a boolean. In the paper [1] the output is a voltage from piezo sensors, but in our model, we use the output of zero instead of the sigmoid activation function and use the output directly. 

\(\rho_{GL} = GL \)

\(\rho_{GR} = GR \)

Angle Sensory Neurons

As we can see in figure 4, the neuron modules in each joint are composed of one sensor neuron and one motor neuron. Whenever the threshold is exceeded, the angle sensor directly inhibits the corresponding motor neuron. The angle sensors are modelled similarly to the receptor sensors with a sigmoid activation function. This leads to the implementation of these equations for the activation functions:

\(\rho_{ES} = (1 + \exp^{\alpha_{ES}(\theta_{ES} - \phi)})^{-1} \)

\(\rho_{FS} = (1 + \exp^{\alpha_{FS}(\phi -\theta_{FS})})^{-1} \)

Motor Neurons

The motor neurons can be modelled as an electrical circuit with a summation of the weighted sensor inputs and a charging and decharging capacitor. This circuit can be implemented numerically, but in this model we use the analytically form of the equation:
\(\tau\frac{dy(t)}{dt} = -y + \sum{w_i\rho_i}\)

Where \(\tau = RC\) the properties of the resistor and the capacitor in the electrical circuit. 

By solving this differential equation, we obtain:
\(y(t) = y_0\exp{-t/\tau} + \sum{w_i\rho_i}\)

\(y\) is here the simulation of the action membrane function for each of the neurons over time. When the charging of the neuron reaches the threshold it is activated and we get an action potential spike before a decharge. We will in this model assume that the activation is instant and we will use \(dt\) instead of the real-time and \(y_0 = y_{t-1}\), as we use a time-step dataset for our simulation. We model the activation of the motor neuron the same way as we did for the sensor neurons, with a sigmoid function, which gives us this equation: 

\(u_{EM{FM}} = (1 + \exp^{\theta_M - y})^{-1}\)

\(u\) is here the output voltage of the neuron. With this we can calculate the voltage for the motors to control the torque applied. 

\(U = M_{AMP}G_{M}(u_EM - u_FM)\)

where \(M_{AMP}\) represents the magnitude of the servo amplifier and \(G_{M}\) is the output gain of the motor neurons in the joint.

Implementation and Results

Sample Set

In order to feed the sample set for a discrete time step and in this way test the neuronal controller, a separate MatLab file "Test.m" was created with a for-loop. The only data for the file that is needed, are the parameters needed in the controller, and the others can be filtered out and not considered. At each step, the data is and plots the time series.

Figure 5: Plot of the motor voltages for the sample data set

As we can see from figure 5, the knee plots have a positive spike right before the flexion of the knee (negative voltage). And it seems that the amplitude of the spike is higher for the left knee than for the right. The hip voltages look, on the other hand, like the example plots provided in Moodle.

Simulation with the multi-body 

Initial Conditions Values
\(\phi_{IC_HR}\) -29 deg
\(\phi_{IC_HL}\) 10 deg
\(\phi_{IC_KR}\)  0 deg
\(\phi_{IC_KL}\)  0 deg
Table 2 : Initial conditions of the multi-body simulation

After implementing the controller into the multi-body model with the initial conditions from table 2, it walks for right under three seconds.

Figure 6 : The motor voltages with the multi-body model

Looking into what parameters that can increase the forward speed, and therefore make the robot walk faster, there are two things that have to be maximized. The force thrusting forward (x-direction) is defined by the torque for stance for an inverted pendulum, in other words, the hip torque. As we want to push forward, this torque needs to be maximized with a stability constraint. In order to do this, we can increase the parameter \(G_{MH}\). For the hip torque to have the maximal effect, the time during which the stance foot is on the ground should be longer. As this stance time is proportional to the difference between the AEP and the \(\theta_{FH}\), this difference should be increased, by for instance decreasing \(\theta_{FH}\).

Figure 7 : Plot of the state space model

As we can see from the state space model, our neuronal controller is not particularly stable and loses stability for every step. When trying to find parameters that affect the forward speed, it only results in the model falling. Even for smaller changes, as well as larger changes, in the hip parameters, such as \(\theta_{FH} +- 1 or +-5 \) or \(G_{MH} +-0.1 or +- 5\) result in an unstable gait, and the model stops running after less than a second. It should be possible to adjust and increase the forward velocity by increasing the \(G_{MH}\) and tuning(\theta_{FH}\) to stabilise the gait. However, with numerous tries of manual tuning, I have yet to find a stable gait with other parameters.

Challenges and Discussion

Firstly, the main challenge in implementing a neural network is to find the right tuning values for weights and thresholds. However, in this assignment, we already had these parameters from the Geng paper [1]. In the beginning, it was a challenge to identify which neuron is doing what during the movement and keep track of right and left, extension and flexion, and hip and knee. Being consistent in the MatLab code with the indices, was therefore necessary. Later, the implementation of the time step in Simulink and MatLab and using the derived time instead of continuous time was a hurdle. There was also a confusion of the convention in the Geng paper of flexion and extension in the hip, which had consequences for the stretch receptor, which I missed for a long time. In addition, trying to find out what causes the spikes in the motor voltages for the knees and how to prevent it from happening, is still something that I can't figure out. And finally finding or tuning the parameters in order to find a stable gait with a different velocity proved to be the greatest challenge of this assignment.

Apart from the challenges, the implementation of a neural network based controller was not very difficult, even without having an in-depth understanding of the entire system. Using the Geng paper [1] and the equilibrium point controller as a starting point, eased the process, but it took some time to get into the already existing models and the different notation. And in the end, a (shortly) stable gait was achieved. However, by manually tuning the parameters, one rapidly risks losing stability. In the case of a desired state output, like for example forward velocity, one needs to find better ways of tuning the parameters to find a stable gait. 

  

References

[1] Geng, T.; Porr, B. ; Wörgötter, F.: Fast Biped Walking with a Sensor-driven Neuronal Controller and Real-time Online Learning. In: The International Journal of Robotics Research (2006); Nr. 25; S. 243 DOI: 10.1177/0278364906063822

Assignment VI - The Spring Mass Walker

Posted by Gyri Reiersen on 11 May 2018, 5:11 PM

Introduction

It is important to understand the mechanics of human locomotion to be able to recreate the same mechanical traits in robots. It has been shown that modelling human walking mechanics is challenging due to the gaits complexity, but running patterns can be however be modelled as a spring-mass system with a defined angle of attack. In this document, we will be presenting a model for the human walking gait based on the equivalent of the running model with a spring-mass system.

Problem specification

The problem is defined as "Program a spring-mass-walker in Matlab/Simulink describe the specific challenges of modelling bipedal walking explore the possible forward speed (maximum instantaneous and average speed)".  As well as developing a walking spring-mass model we will discuss challenges of modelling bipedal walking and look into a way of finding it's parameters. Although stiff legs are often used to model the dynamics of human walking we will in this assignment develop a model based on the results of the paper published by Geyer [1]. Finding the right model for walking is difficult, and one has earlier used models with stiff legs connected to a mass, simulation the human body's centre of mass (CoM), which is a simpler approximation. However, this model type is not able to describe the forces in the double support phase, when the two feet are both in contact with the ground. As we can see from Geyer's results simulating two compliant mass-less legs in a spring-mass model, the characteristic stance dynamics is incorporated and gives better results. The spring-mass walker is, however, a more complex model.

 Figure 1: Illustration of the spring-mass model taken from Geyer [1]

 

The parameters used for the spring-mass walker are;
Length of the springs \(l_0\)
Spring stiffness \(k\)
Mass \(m\)
Angle of attack \(a_TD\)
Foot point position at instant i \(FP_i\)

In Figure 1, we see how the spring-mass model is divided into four phases. There are two types of support phases, single (one spring in contact with the ground) and double (both springs in contact with the ground). With this SMW model it is possible, like shown in the figure, to model bipedal walking, also for the forces during double support, however, the challenge is to find the right parameters. In the next part, we will develop the model.

Developing the SMW model

In order to model the dynamics of human walking, we need to start with the same characteristics that of a human. Firstly, the mass of an average human is 80kg and leg length of 1m, we can find the right stiffness of the springs. Further, we need to find the right angle of attack, which is defined as the angle between the spring and the ground at touch down (TD). To achieve a stable model, you would also need to find appropriate initial conditions, such as the height \(y_0\) and the velocity \(\dot(y)_0\). Without the right parameters, this model is highly unstable and would fall shortly after initialization.

We can interpret the SMW as the sum of two subsystems, one for each spring, that overlap in the double support phase. The Simulink model for such a subsystem was provided by the professor as a model for a spring-mass runner with one leg. This running model is a good starting point for developing the walking model.

 

Figure 2: The Simulink block for a single spring

 

In figure 2, we see the foot point position of the spring \(xFp\) and the position of the mass are used to compute the current spring length and further the forces applied to the mass by this spring. By using two of these spring force blocks we can model our SMW in Simulink.

Figure 3: The entire Simulink model for the SMW

In order to simulate the bipedal walking, we need to create a condition for triggering or enabling the overlapping of the two springs, the double support phase. This is called the touchdown (TD)/take-off (TO) conditions. In this model, we have chosen to trigger at the touchdown.

Figure 4: The Simulink block for the triggering of the double support phase

As we can see from figure 4, the main idea is that when the forward spring is at TD \(y<l_0*sin(a_TD) \) and the mass is going down towards the ground \(\dot(y)<=0\) it takes one step, both springs are on the ground, and the double support phase is triggered. The assumption that the angle of attack is constant throughout the simulation is very important for the computation of the new foot point.

Figure 5: The Simulink block for the changing of the foot point position

Figure 6:  The Simulink block for calculating and storing the foot point position at TD

The triggering TD triggers the double support phase and also the block that calculates the new foot point position of the spring shown in figure 6. In figure 5, we use a flip-flop to switch between the two spring subsystems and to avoid that the model just skips with the same leg always in front.

Excluding all perturbations, the forces of the total system are conservative. We have assumed in this model that we should have a total energy of 810,7 J, which is considered as the real energy for the human walking dynamics.

In order to complete the SWM we also need to derive the movement of the CoM over time from the forces that are applied to it. We do that by computing the equations of motion for a two-dimensional space (both in x- and y-direction).

\(m*\ddot(x) = \sum(F_x) \)

\(m*\ddot(y) = \sum(F_y) \)

 For the two directions, horizontal and vertical, the three forces that are applied to the mass will be the two springs and the force of gravity (only applied in the vertical direction). This leads us to the solution.

\(m*\ddot(x) = Fx_S1 + Fx_S2  \)

\(m*\ddot(y) = Fy_S1 + F_S2 + F_g \)

By defining \( F_si = (l_0 – l_i)*k \) and \( l_i = \sqrt{(x_FPxi - x)^2 + y^2} \) and developing further, we get:

\(m*\ddot(x) = \frac{x_FPx1}{dl_1}*F_s1 + \frac{x_FPx2}{dl_2}*F_s2   \)

\(m*\ddot(y) = \frac{y}{dl_1}*F_s1 + \frac{y}{dl_2}*F_s2  + g*m \)

As you can see in figure 3, we use these equations to compute the movement of the SMW over time.

Discussion

We have now seen that the SWM can be used to imitate the human walking gait. However, it has its limitations and can be improved. As a part of the assignment, we need to look into the possible forward speed (maximum instantaneous and average speed. Looking at our results for the stable model, we see that with the initial horizontal velocity of 1,06 m/s, the speed changes throughout the walk with a maximum of 1,29 m/s and an average of 1,16m/s.

 

Figure 10: Plot of the velocity for an initial horizontal velocity of 1,06m/s

Figure 10: Plot of the velocity for an initial horizontal velocity of 1,22m/s

 

By playing around with the model, only changing the initial horizontal velocity, it is possible to get a nearly stable model for a maximum of around 1,22 m/s as initial velocity. This model will be unstable in the beginning but will stabilize for some time as seen in figure 11. The average horizontal speed will in this case also be increased slightly to 1,17 m/s, but it seems to stabilize at the same speed as the more stable model.

 

Figure 11: Phase plot of the stable model with an initial velocity of 1,06m/s

Figure 12: Phase plot of the unstable model with an initial velocity of 1,22m/s

 

 

Finding the parameters that allow our model to be stable is the greatest challenge of modelling a bipedal walking using the SMW model. Although there are some boundaries to your parameters to start with the parameter space is limited as discussed by Geyer [1]. The assumptions made in the beginning for the mass and leg length corresponds to an average human, but for the other parameters need to be found through testing. This tuning of parameters is time-consuming and complex, but one solution to this would be to use artificial intelligence such as a neural network.
